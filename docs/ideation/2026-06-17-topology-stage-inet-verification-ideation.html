拓扑阶段的 INET 验证：把库里的拓扑发出去验对不对

日期 2026-06-17 · 主题 topology-stage-inet-verification · 模式 repo-grounded

聚焦：把已生成在 SQLite 里的拓扑记录，以什么格式、走什么渠道发给 INET（OMNeT++ TSN 仿真）做验证——同时验拓扑图本身和 MAC 转发表；且 MAC 转发表目前还没存进数据库。

一句话背景：现在拓扑画完之后，软件几乎不知道它"对不对"——唯一的检查只看了"节点数 > 0"和"有没有连到空气的线"。要真正验证，得把拓扑送进 INET 看它能不能跑、转发路径对不对。本次梳理给出 7 个可深入的方向。

两个被核实的关键事实：①「MAC 转发表没存库」属实——但其实代码里早就能用 BFS 现算出一张转发表（build_artifacts 里的 build_legacy_mac_forwarding_table），只是算完没保存。② 拓扑节点存的类型是旧的 Qunee 画图类名，不是 INET 认识的模块名——任何"发给 INET"都得先补一张类型对照表。

Grounding Context · 代码现状

库里的拓扑长什么样：只有两张表真正被写——topology_nodes（节点：编号 imac、名字、类型 switch/endSystem/server、坐标）和 topology_links（链路：两端节点、端口号塞在 styles_json 字符串里、速率）。端口没有结构化成列，速率/队列等更深的字段没落库。

MAC 转发表 / 各类节点配置表（nodes_array_cfg(fwd_cfg)、nodes_object_cfg(static_mac_cfg) 等）在表结构里定义了，但没有任何生产代码往里写——只有测试和导入回环碰过。所以 boss 说的"MAC 转发表没生成"成立。但 topology_compute.rs:2438 的 BFS 已经能现算出一张转发表（每个交换机 × 每个目的节点 → 出端口），随 build_artifacts 工具吐成 mac-forwarding-table.json，用完即弃、不落库。

现有"验证"：MCP 的 validate 工具只查"节点数 > 0"和"悬空链路"。没有任何 INET / 语义级验证。

历史：曾经有一套 INET 导出 + smoke 仿真的计划（05-21-001），整套导出器已被删除；但 docs/ned-contract.md 还留着完整的 NED 导出契约（TsnSwitch / TsnDevice / EthernetLink、INET 4.6、命令 inet -u Cmdenv -f omnetpp.ini -n .）——文档与代码已漂移。

通道与信任模型：拓扑 sidecar 是进程内的 127.0.0.1 HTTP + 令牌；MCP 工具只是转发给它。今天没有任何进程拉起 INET，远程 API 客户端已删。信任上：worker 只认 MCP 返回里的 mutationId，agent 没有写权限——新验证闸应该是"一条 sidecar 路由 + 一个工具"的同款形状，不要重建那个被删掉的"让大模型回传结果"的协议（多一个事实源正是当年被删的原因）。

Topic Axes · 五个验证面

A 验什么：拓扑图有效性（连通/端口配对/链路对称/角色约束/编号重复）vs MAC 转发表有效性（可达/无环/路径全覆盖/与拓扑一致）。两个不同目标。
B MAC 表从哪来：现在没存——现算现用（已有 BFS）/ 存库再验 / 让 INET 自己生成再对账。先有数据才能验它。
C 发出去长什么样：INET 原生的 NED+INI+XML / TSNsched 式 JSON / 旧的 build_artifacts JSON / 专门的转发表 XML。
D 怎么发、在哪跑：进程内 sidecar 拉本地 INET / 远程仿真主机（HTTP 或 SSH+SCP）/ MCP 工具 / worker 拉起。
E 绿勾代表什么、结论怎么回传：纯结构检查（不跑仿真）/ INET "能加载运行" smoke / 真正的可调度性（NC/SMT）；口径标注；结论经可信 sidecar 路由回传。

1

零仿真结构校验器（先落地、不依赖 INET）

面 A · 验什么 把握 90% 复杂度 低
说明

大部分"拓扑对不对"根本不用跑仿真，是纯图论：连通性、端口两两配对、链路对称、有没有孤立节点、交换机/端系统角色是否合法、编号是否重复。加一条 sidecar 路由 /db/topology/verify，直接读库里的 topology_nodes+topology_links，把这些检查做完，返回 {ok, errors[], signalId}。检查规则做成一张"声明式规则清单"（数据，不是写死的 if），以后加规则就是加一行。这一半立刻能上、随时能跑、断网也能跑，是把"拓扑验证"拆成"先结构、后仿真"里能马上交付的那一半。

依据 direct

现有 validate 只查节点数+悬空链路（topology-tools.ts）；图数据已完整落库；前序梳理已定调验证闸形状为 /db/<stage>/verify → {ok,errors[],signalId}。规则清单形状来自跨域类比（芯片设计的 DRC 规则库与检查引擎分离）。

为何重要

它把"拓扑验证"里最便宜、最确定的一半立刻变现，且不被 INET 环境拖住；INET 只留给图论答不了的问题。

代价

结构对 ≠ 时延/调度对——必须靠口径标签（点子 2）说清"这只是结构级"，否则会被误读成"已验证通过"。

2

一句人话结论 + 口径标签

面 E · 绿勾代表什么 把握 85% 复杂度 低
说明

不管后台做的是结构检查、INET 加载 smoke 还是更深的验证，回给用户的永远是一句中文结论（如"拓扑连通、端口配对正确，可用"或"交换机 SW3 有一条线没接对端，拓扑不完整"），仿真器的原始输出收进可展开的细节里。同时每个结论强制带一个口径标签（仅结构 / 仅能加载运行 / 已验可调度性），并把工具命名为 simulation_loadability_check 而不是含糊的 tsn_verify——让"绿勾"在界面上永远不可能脱离它的适用范围单独出现。

为何对 boss 关键

boss 看不懂工程术语——一份看不懂结论的验证功能等于没用。"一句人话"是真正的交付物，仿真只是它的来源。

依据 direct + external

前序梳理点子 #5 已要求口径隔离（强制 coverage 字段、命名 simulation_loadability_check）；外部 DO-178C 分级保证、"仿真是统计下界不是可调度性证明"。现在（planning-export 还没建）是钉死这套命名最便宜的时机。

代价

需要约定一套结论文案模板；多个级别的口径词要提前统一，否则后面四个阶段各说各话。

3

MAC 转发表现算现验、不落库

面 B · MAC 表从哪来 把握 80% 复杂度 低-中
说明

MAC 转发表是拓扑的纯函数——已有的 BFS（build_legacy_mac_forwarding_table）就能从节点+链路算出来。所以不必先存库再验：验证路由在同一次调用里现算这张表，并直接校验算出来的结果（每个目的地恰好一个出端口、不指回源头形成环、所有可达端系统都覆盖、出端口确实在该交换机上），算完即弃。等于把"该不该给数据库加 MAC 列/写入器"这个争论整个绕过去——数据已存在，验它的计算结果就好，数据库保持单一事实源。

依据 direct

topology_compute.rs:2438-2475 的 BFS 已现算每交换机×每目的的出端口、随响应吐出、从不落库；那些 MAC 配置表零生产写入者。换句话说：表能算，只是没存。

为何重要

直面"MAC 表没生成"的真相，并选了改动最小的解法（现算现验），契合"不留冗余、单路径"的信任纪律。这是"验 MAC 表有效性"这半最便宜的入口。

代价

如果未来有真实消费者（真机/规划引擎）确实需要持久化的 MAC 表，那是另一个由消费者驱动的决定，不在本验证范围里。

4

让 INET 自己生成转发表，再跟我们的对账（INET 当裁判）

面 A/B · 验什么 + 表从哪来 把握 75% 复杂度 中-高
说明

与其由我们断言"这张转发表对不对"（需要先定义什么叫对），不如让 INET 用它自带的转发表配置器从拓扑自动生成一张 FDB，再把它和我们 BFS 算出来的那张逐条对账：一致 = 我们的算法和一个成熟参考实现吻合；不一致 = 精确定位到哪个"交换机×目的"的出端口对不上（像 terraform plan 那样给出逐项差异，而不是一个 pass/fail）。这同时化解了"MAC 表对谁有效"的悬而未决——把 INET 自己当作那个权威消费者。

依据 external + direct

INET 可用 *.macForwardingTableConfigurator.typename="" 切换自动/静态 FDB；"期望 vs 实际"对账模式来自 Cumulus NetQ / 西门子 PRONETA；我们的 BFS 表已存在可供对账。6 个生成视角中有 6 个独立收敛到这个方向，是最被反复指向的点子。

为何重要

它才是 boss 真正问的"发给 INET 验 MAC 转发表"，且把"对谁有效"这个隐藏假设变成了屏幕上可见的差异表。

代价

依赖能跑起来的 INET（见点子 5、7 的前置）；对账口径要明确（INET 走最短路，我们的 BFS 也走最短路，规则要对齐）。

5

节点类型→INET 模块 的对照表：先建这块砖

面 C · 发出去长什么样 把握 80% 复杂度 中
说明

库里节点存的类型是旧的 Qunee 画图类名（Q.Graphs.exchanger2 / node / server），不是 INET 认识的模块（TsnSwitch / TsnDevice）。任何"发给 INET"都必须先过这道翻译。把它做成一张带版本号的对照表（数据资产），并升格为一项验证检查：任何节点的类型在表里找不到对应，就是一个 unmappable_node_type 错误，在 INET 还没启动前就明确报出来（而不是生成一个坏掉的 NED 文件、让 INET 抛一句看不懂的崩溃）。还可加一道"导出→重新解析→比对"的往返检查，确保翻译无损。

依据 direct

调研确认存的 sync_type._classPath 是 Qunee 类、非 INET NED 模块，"任何 INET 导出都需要一张新的类型映射"。这是所有 INET 路径共同的前置长杆。

为何重要

把一类"INET 莫名加载失败"的隐性故障，前移成一句精确、可行动的应用内报错；同时这张翻译表是后面每个碰 INET 的阶段都要跨的边界，做成单一资产可复用。

代价

需要有人确定并维护这张对照表；server 类型映射到哪个 INET 模块需要确认。

6

序列化即交付物：一个 inet-bundle 验证与导出共用

面 C · 发出去长什么样（杠杆） 把握 65% 复杂度 中-高
说明

定义一套序列化产物——{network.ned, omnetpp.ini, fdb.xml, manifest.json}——验证时用的就是这个包，不另造一份临时格式；将来 planning-export 真要导出时，沿用同一套布局（只在 ini 里加 GCL/流），而不是再发明一个并行格式。manifest 里钉上 {格式版本, 来源 mutationId, 口径} 作为统一的来源头。配套：检查器与生成器解耦（CI 里用一个已知好/已知坏的样例分别断言"必过/必拒"），并把还留在磁盘上的 docs/ned-contract.md 从"漂移的死文档"升级为生成器输出要满足的可执行契约。

依据 direct + reasoned

旧的导出器（ned/ini/gcl 等多文件）已删除、docs/ned-contract.md 仍在磁盘描述着不存在的目录（漂移）；这是四个阶段验证里的第一个，现在选定的格式会被后三个阶段继承——此刻钉死格式是最便宜的时机。

为何重要

最贵、最难改的就是被多个下游依赖的序列化格式。趁只有拓扑用它时定下来，避免"验证用一份、真导出又一份"的重复，也把唯一幸存的契约文档变成资产而非负债。

代价

需要先想清楚 export 阶段的需求，否则"通用格式"可能猜错；属于偏前期投资，单独看本期收益不如点子 1-3 直接。

7

远程仿真主机 + 阶段无关的验证通道

面 D · 怎么发、在哪跑 把握 70% 复杂度 高
说明

INET/OMNeT++ 是个重型本地依赖，不太可能塞进桌面应用。让它跑在远程仿真主机上：sidecar 把序列化好的包通过 SSH/SCP 送到一个按次隔离的远程目录，用结构化参数（不是拼 shell 字符串）跑 inet -u Cmdenv -f omnetpp.ini -n . 的加载 smoke，带超时、进程组杀、密钥脱敏，结果经可信 sidecar 路由回来。关键设计：这条路由从第一天就做成阶段无关 /verify/inet（带 stage 参数），让后面 time-sync/flow/export 复用同一套"导出+跑+回收"管线，而不是各建一条。

依据 direct

前序梳理已提"阶段无关 /verify/inet"；被删的 05-21-001 留下了可复用的跨机纪律（SSH/SCP 按次目录、结构化参数、超时、进程组杀、脱敏）；今天 worker 是唯一的外部进程拉起点。

为何重要

决定了 INET 物理上在哪跑——本地-only 会把大多数没装 OMNeT++ 的用户挡在门外、让功能形同虚设。通道做成阶段无关，则后三个阶段只需加一套模板而非重建管线。

代价

本期最复杂的一条，涉及外部主机/认证/网络——属于改变数据流的架构决定，需要 boss 拍板（要不要远程、主机谁提供）。

Rejection Summary · 落选与原因

#点子落选原因
1把现算的 MAC 表持久化进库再验是"MAC 表从哪来"这道分叉的另一边；现算现验（点子 3）改动更小、不引入会失同步的派生数据。作为备选记录。
2按每个消费者建 Pact 式契约验 MAC 表规划引擎已不消费 MAC 表、真机待定，多消费者验证器属过度设计；并入点子 4（INET 当裁判）+ 点子 2（口径）。
3每次拓扑改动就增量跑、带缓存（verify-on-write）过早优化；结构检查本就便宜，仿真缓存等真接了 INET、有性能问题时再说。
4完整 DO-178C 分级阶梯（含 not_evaluated 顶层）方向对，但本期只实现底部两级；并入点子 2 的口径标签即可表达"未做的更高级别"。
5opp_test 指纹/golden-file 回归是序列化器的回归手段，并入点子 6 的检查器/生成器解耦。
6对转发表做属性测试（无环/全可达）是验证手段而非独立方向；其断言已并入点子 3 的校验项。
7"导出→重解析→比对"的往返不变量单列是点子 5（类型映射）的配套检查，已并入，不单列。
8把验证结果做成大模型回传的结构化协议正是当年 05-21-001 被删的"多事实源"反模式；所有点子都改走 sidecar 单路由+可信信号，故不保留。
