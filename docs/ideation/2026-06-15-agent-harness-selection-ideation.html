Ideation：为 TSN Agent 选 agent harness —— 替代 Claude Code

本地部署 Qwen3.6 的前提下，用什么成熟开源 harness 驱动"工具型领域 agent"，既能嵌入 Tauri 分发，又不靠脆弱的"提示词+自写 loop"。

日期 2026-06-15 主题 agent-harness-selection 模式 repo-grounded 约束 开源 · 本地模型 · 可嵌入 · 成熟 loop · 不自写 loop

Grounding Context

本项目 agent 的真实画像是一个"工具型领域 agent"——模型 + 8 个 topology MCP 工具（topology_initialize / inspect / apply_operations / validate / build_artifacts …，后端全走 HTTP sidecar）+ SKILL.md 领域指引（TSN 拓扑），多轮、工具密集；不是通用 coding agent，不需要 Read/bash/web。

必须满足的集成点（来自 worker 代码核实）：① 工具调用协议——{id,name,args,phase,status,result} 透传 + id 去重；② SKILL.md 注入 system prompt（三段 sentinel 拼接，harness 无关，任何能接受自定义 system prompt 的 harness 都能复用）；③ 流式 tool_call 事件喂前端折叠卡片；④ 三态 fail-closed 校验 + sanitizer/audit 边界（业务产物绝不进 log、大结果绝不回流 prompt）；⑤ 接本地 OpenAI 兼容端点；⑥ 模型 pin，不继承个人设置。

关键切口已现成：worker 的 runClaude(prompt, options, queryFn = query) —— query 本就是可注入参数；agent-types/agent-adapter/sanitizer 是稳定进程边界，当初设计流式卡片时就刻意留了"客户端无关事件契约"为换后端铺路。换 harness = 只替换 worker 内部 query 调用 + for await 消息循环，领域逻辑整体保留。

当前 Claude Agent SDK 的负担：208MB native binary（打包/分发噩梦，本次一连串 Windows 崩溃的根源）；~/.claude 认证强依赖（无法分发）；SDK 内部 spawn MCP 子进程、失败无恢复。

跨 harness 必须保住的硬不变量：正确性靠代码层强制（工具白名单 + trusted-signal 提取器）而非 prompt；写库动作一步闭环，不依赖模型多步编排（换到能力弱的本地 Qwen 时这条更要命）。

Topic Axes

  • 嵌入与分发 — 运行时/语言/体积：能否作为库塞进 Tauri 的 Node worker（而非整体 binary）
  • 本地模型 + 工具可靠性 — 接 OpenAI 兼容端点 + guided/structured 工具调用兜底（弱模型能否稳定调工具）
  • MCP 与域逻辑复用 — 能否直接复用现有 topology MCP server，少改域逻辑
  • 迁移风险与切口 — 改动面大小、可逆性、对既有不变量的保护
  • 模型韧性 — 是否留可换/可对比模型的缝，降低单押注 Qwen 的风险

Ranked Ideas

方案速览（点击跳转）

I1

窄切口迁移：只替换 worker 内部的 query + 消息循环

direct置信 90%复杂度 Medium
Axis

迁移风险与切口

方案

不重写整条 runtime。保持 agent-types 契约、agent-adapter 三态边界、sanitizer/audit、客户端无关 tool_call 事件契约、以及独立的 topology MCP server 全部不动；只把 worker 内部 runClaude 里的 queryFn = query（Claude SDK）换成新 harness 的调用，并把那个 for await (const message of …) 循环按新 harness 的事件流重写。

Basis

direct: worker.mjs:32 runClaude(userPrompt, options, queryFn = query) 已把 SDK 入口设计成可注入参数；记忆明确记录"客户端无关工具事件契约是为换 Qwen 后端刻意设计的抽象缝"。

为何关键

这是其余所有方案的地基——它把"换 harness"的爆炸半径锁死在 worker 一个文件内，领域逻辑（topology/sidecar/三态/卡片）零改动。是确定性最高、风险最低的元决策。

代价

新 harness 的工具暴露面、env 注入、会话续接（resume 无等价物，需用已有 buildConversationContext 喂多轮）要逐项对应；summarizeSdkOptionsForAudit 需按新 option 形态重写。

I2

Vercel AI SDK v6（ai）作为 harness —— 首选

direct置信 85%复杂度 Medium
Axis

嵌入与分发 / 本地模型 + 工具可靠性

方案

用 TypeScript 的 Vercel AI SDK v6 当 harness：createOpenAICompatible(baseURL) 直指本地 vLLM/SGLang 端点；ToolLoopAgent（内置 stopWhen）跑成熟的多轮工具循环；@ai-sdk/mcp 稳定版直接挂现有 topology MCP server；experimental_toToolCallRepair 在工具调用失败时自定义修复（含切更强模型重生成参数）。

Basis

external: AI SDK v6（2026，月下载 2000 万）原生提供以上能力，是 TS 生态唯一带原生 tool call repair 的 harness；direct: worker 本就是 Node，TS harness 嵌入 Tauri Node sidecar 最干净（整体打包 30–80MB，对比 Claude Code 208MB binary）。

为何首选

同语言（TS/Node）零跨运行时摩擦、体积最轻、loop 成熟（满足"不自写脆弱 loop"）、原生 MCP 复用域逻辑、且独家 repair 给弱模型工具调用兜底。约束命中度最高。

代价

比 mastra 少一层 agent lifecycle 抽象，事件流/卡片对接要自己组织（但有 ToolLoopAgent 兜底，不是"裸写 loop"）；v6 较新，需锁版本跟踪 API。

I3

推理层 XGrammar guided decoding —— 工具可靠性的第一道防线

external置信 88%复杂度 Low
Axis

本地模型 + 工具可靠性

方案

工具调用可靠性的主力不在 harness 层，而在推理层。vLLM / SGLang 已把 XGrammar 设为默认结构化解码后端——只要 harness 把工具的 JSON Schema（response_format: {type:"json_schema", …}）传给本地端点，参数就在 token 生成时被强约束，非法 token 直接屏蔽。harness 的 tool_call_repair 只是事后兜底。

Basis

external: XGrammar-2（2026，单 token <40µs）已是 vLLM/SGLang 默认；推理侧约束与 harness 解耦——这是本项目"35B MoE 能否稳定调 8 个 topology 工具"的决定性好消息。

为何关键

它把"弱模型工具可靠性"这个最大风险，从"赌 harness/prompt"变成"推理层硬保证"。配 harness 层 repair 形成两道防线，类比类型系统（编译期约束）+ 运行时异常处理。

代价

需在本地推理服务侧正确开启/配置 guided decoding，并保证每个工具的 JSON Schema 干净（现有 Zod schema 可转）；约束解码对超复杂 schema 有少量延迟。

I4

先做 Qwen3.6 能力 spike，再定 harness

reasoned置信 85%复杂度 Low
Axis

迁移风险与切口

方案

在投入 harness 迁移工程前，先用本地 Qwen3.6 + 一个最小 OpenAI-兼容 tool loop（或直接 AI SDK 快试）跑通一遍标准 topology 流程（对标 generic-line / dual-plane 真机验收点），重点压测：多步工具编排、零参工具首调用、确认/阶段推进这类靠文本判定的环节。

Basis

reasoned: 记忆明确——本项目大量依赖"模型听话"（多步 MCP 编排、正则纠偏自然语言、零参工具），而 Qwen3.6 35B 能力大概率弱于 Claude Sonnet，这些正是换弱模型后最先崩的点。先验证可行性，避免在错误模型上盖楼。

为何关键

harness 选型再优，模型调不动工具也是空中楼阁。这个前置 spike 用最小成本回答"Qwen3.6 + XGrammar 到底能不能稳定跑这套 8 工具流程"，结果直接决定后续投入与 harness 细节。

代价

spike 本身要一套最小循环（一次性、可丢），与"不自写生产 loop"不冲突——它是验证而非生产代码。

I5

mastra 作为 harness —— 框架化次选

external置信 72%复杂度 Medium
Axis

MCP 与域逻辑复用 / 嵌入与分发

方案

用 TypeScript 原生的 mastra（底层基于 AI SDK）：MCPClient 连现有 topology MCP server、MCPServer 反向暴露；Zod 定义 tool schema、类型安全；内置 trace/eval/observability，一个框架同时管 agent lifecycle + MCP + 可观测。

Basis

external: mastra（TS 原生、2025 末融资快迭代）提供双向 MCP 桥 + 内置工具编排，对"工具型 domain agent + 复用现有 MCP server"契合度极高。

为何次选

比 I2 多一层"全家桶"抽象（lifecycle + observability 内置）。若希望少自己拼装、要现成的 trace/eval，选它。

代价

API 仍在变动、无 SOC 2，引入需锁版本；对"单 agent 工具型"场景，它的上层抽象可能偏厚（继承 AI SDK 的 repair 而非自带 grammar 兜底）。

I6

LLMProvider 抽象层：Qwen/Claude 双接、可对比可回退

directreasoned置信 70%复杂度 Medium
Axis

模型韧性

方案

在 worker 内把 LLM 调用收敛成一个 LLMProvider 接口（沿用已有 queryFn 注入缝），同一 harness 下既能指向本地 Qwen 端点、也能指向 Claude/其他，配置可切。

Basis

direct: queryFn 已是注入点、事件契约已客户端无关；reasoned: Qwen3.6 能力存疑（见 I4），留一条"质量不够就回退 Claude / 实测对比"的缝，避免单押注一个未验证模型。

为何有价值

把"选哪个模型"从一次性赌注变成可逆、可 A/B 的工程选项——尤其在 Qwen3.6 真实能力未实测时，这层韧性很划算。

代价

不同模型的 tool 协议/system prompt 调优有差异，抽象不能太薄；过度通用化有 YAGNI 风险——按"Qwen + 一个回退"两条实现即可，不要预先支持五家。

I7

@openai/agents JS —— 轻量备选

external置信 60%复杂度 Medium
Axis

嵌入与分发

方案

OpenAI Agents SDK 的 JS 版：Agent/Runner/Tool 原语简洁，setDefaultOpenAIClient(new OpenAI({baseURL})) + setOpenAIAPI('chat_completions') 接本地端点，MCP 走 stdio/SSE/HTTP。

Basis

external: 官方 TS、原语清晰、可接本地端点 + MCP；体积控制好。作为"mastra 嫌厚、AI SDK 想要更结构化原语"时的中间选项。

为何备选而非首选

无 grammar 层兜底（靠 I3 推理层补）、MCP session 生命周期要自管；生态成熟度与 repair 能力不及 AI SDK。

代价

工具可靠性更依赖推理层 XGrammar；MCP/会话管理要多写胶水。

倾向性推荐

主线：I1（窄切口）+ I4（先 Qwen spike）+ I3（推理层 XGrammar）作为地基与前置，harness 首选 I2（Vercel AI SDK v6），并以 I6 的 LLMProvider 抽象保住可回退缝。

理由：同为 TS/Node、嵌入最轻、loop 成熟（满足"不自写脆弱 loop"）、原生 MCP 直接复用 topology server、独家 tool repair 兜底；真正的工具可靠性靠推理层 XGrammar 硬保证。mastra（I5）作为"想要全家桶 + observability"的平替随时可换——两者底层同源，切换成本低。明确弃用 goose（binary 不可嵌入）与 Qwen Code SDK（实验性、接不了本地端点）——这两个正是你最初点名的，调研结论是都不合适。

决策与执行路线

本 ideation 主线（去 Claude Code、迁本地 Qwen + Vercel AI SDK）是中长期方向；但 Windows 打包 app 当下完全跑不起来——根因是打包链没把 Claude Code 的 208MB native binary 带进 release bundle，SDK 运行时找不到 claude 可执行，agent 直接无返回（mac 一直只跑 dev 模式、node_modules 里有 binary，所以从没暴露）。需要一个能立刻验证打包态 agent 可用的短期桥。经讨论定两阶段：先修可用，再做迁移。

短期 A（当前动作）：往发布包加 Claude Code binary

把平台对应的 claude 打进 tauri resources，worker 用 pathToClaudeCodeExecutable 指向它，让打包态 agent 立刻能跑。临时桥：加重 Claude Code 依赖、+208MB/平台、仍需用户侧 ~/.claude 认证（仅自用场景成立）。

中长期 B：本 ideation 主线

窄切口换 harness（Vercel AI SDK v6）+ 本地 Qwen3.6 + 推理层 XGrammar。去掉 208MB binary 与 ~/.claude 依赖，包瘦回几 MB、可分发、数据不出境。

A 的操作清单（4 处改动）：① CI（production-build.yml）每平台 build 前把 node_modules/@anthropic-ai/claude-agent-sdk-{平台}/claude* 复制到固定路径（Windows 是 claude.exe）；② tauri.conf.json 的 resources 带这个 binary；③ commands.rs resolve 打包 binary 路径（复用 verbatim strip）注入 worker payload；④ worker 读出后设进 sdkOptions.pathToClaudeCodeExecutable。本地 mac 先验证机制，再 CI 出 Windows 包真测。

Rejection Summary

候选淘汰原因
goose（Block，Rust）是 CLI/桌面 app 整体 binary，不是可嵌入的库，塞不进 Tauri Node worker（正迁 Tauri 但仍是 binary）——你点名的候选，实测不合适
Qwen Code SDK（"qwenpaw"）TS SDK 仍实验性，且 baseURL 未暴露、接不了本地 vLLM/SGLang 端点——你点名的候选，对本地 Qwen 反而不可用
LangGraph（JS / Py）状态机图模型概念重，单 domain 工具型 agent overkill；TS 版文档/成熟度落后 Python 约半年（11x 团队已弃用转 mastra）
Pydantic AI（Py）结构化输出最强，但 Python 运行时——嵌入 Tauri 需 Python sidecar，带 runtime + pip 依赖，冷启动慢、分发复杂
Strands Agents（AWS，Py）生产成熟但 Python sidecar 分发成本同上；本项目无 Python 后端基建
smolagents（HF，Py）Python-only + 无原生 MCP，code-as-action 定位不符
CrewAI（Py）角色扮演多 agent 范式，Python-only，不适合单 domain 工具型
AutoGen / AG2（Py）对话辩论范式；微软主版置维护态，AG2 为社区 fork
Semantic Kernel（.NET/Py）偏企业 Azure/.NET 生态，与并入 Microsoft Agent Framework，运行时不符
LlamaIndex定位 RAG/检索 pipeline，不是 agentic loop harness
Agno / Rivet / xsAIPython 框架 / 可视化 node 编辑器 / 资料过少——均不契合"可嵌入工具型 TS harness"
