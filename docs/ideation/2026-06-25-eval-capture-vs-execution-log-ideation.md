# 执行日志模块 vs 大模型交互 eval 采集：合并、替换还是废弃

- 日期：2026-06-25
- 模式：仓库内分析（repo-grounded）
- 焦点：从「存放位置 / 存放内容 / 使用场景」三个角度对比现有执行日志模块和"保存所有大模型交互做 eval 素材"的需求，判断**能否合并 / 是否直接废弃现有日志模块**
- 产出：6 条排序方向（约 40 个原始候选，5 个发想 agent 跨 5 轴生成后去重）

## 一句话结论

**不要合并。** 现有执行日志模块和 eval 采集是**两个需求相反的关注点**——在你说的三个角度上全部对立（见下表）。把 eval 硬塞进现有日志管道，每个角度都要开例外，是三个选项里风险最高、收益最低的。

正解：**新建一条 eval 采集**（在 worker 的 SDK 边界采**原文**、落**独立、不随会话删的 store**、自动带 verification 弱标签和 prompt 版本指纹），**现有执行日志模块可以直接退役**（你已说它对你没用，符合项目"默认单路径、拒绝冗余"的偏好），或降级成新原始层的一个派生视图。**旧日志数据不迁移**（已截断脱敏，对 eval 不可恢复），从切换点起干净采集。

## 三个角度的对立（这就是"不能合并"的根据）

| 角度 | 现执行日志模块 | eval 素材需求 | 结论 |
|---|---|---|---|
| **存放内容** | 事件时间线（"用户提交需求"只存预览、"已保存"、错误）；**截断**（msg 480 / details 6000）+ **脱敏**；不是模型原始输入输出 | raw prompt（骨架+SKILL.md+上下文）+ **未截断**响应 + 完整 tool calls + 标签 | 维度正交，不是详略之差 |
| **存放位置/保留** | `~/Library/Application Support/com.tsnagent.app/logs/sess-<id>/agent-run-<run>.jsonl`；**10MB/会话上限**；**删会话即删整个目录** | append-only、**无上限、删会话不删**、可长期复利累积 | 保留策略直接相反 |
| **使用场景** | 开发者翻抽屉排障（DiagnosticsDrawer，**连导出按钮都没有**） | 离线 eval harness / 喂模型；**导出成 JSONL 数据集** | 消费方和动作都不同 |

> 真正的对话内容其实在 `sessions.payload`（messages/toolCalls/verification），但同样是**脱敏+截断的显示态、且随会话删除消失**——拿它当 eval 源是诱人但错误的捷径。

## 拆解的 5 个轴

- **A. 存放内容** — 事件时间线 vs 原始交互对（prompt/响应/工具/标签）
- **B. 存放位置/格式/保留** — 文件/payload/独立 store；上限与删会话即删
- **C. 使用场景/消费方** — 排障查看 vs eval 数据集导出/回放
- **D. 采集时机/管道** — 何时落、截断脱敏与否、谁写（worker/adapter/sidecar）
- **E. 迁移/退场** — 合并 vs 替换 vs 废弃旧模块

## 排序后的方向

### 1. 采集点下沉到 worker 的 SDK 边界（地基，先定这个）

- **轴**：D　**信心**：高　**复杂度**：M
- **是什么**：只有 worker（`claude-agent-worker.mjs`）拼 prompt（`SYSTEM_PROMPT_SKELETON` + 注入的 SKILL.md + 上下文）、调 SDK、拿到响应的那一刻，能看到**未截断、未脱敏的原文**。在那里加一个"原始交互 tee"，先把 request+response 原样写入 eval store，**再**走现有 redact/truncate 产出 UI/会话的显示态。
- **依据**：`direct:` 现在是 `agent-adapter.ts` 在 run 之后补记一条 agent 诊断，那时原文已被加工/丢弃；payload 也是显示态。
- **为什么是地基**：采集点选错，后面全废——任何在 worker 边界之上（DB/Tauri 层）采的数据都已经有损，eval 再也补不回原文。所有其它方向都建立在"原文采到了"之上。

### 2. 独立 eval store，生命周期与会话彻底解耦

- **轴**：B　**信心**：高　**复杂度**：M
- **是什么**：eval 数据落到**不挂在 `sess-<id>/` 下、不设 10MB 上限、删会话不删**的独立位置（独立目录的 append-only JSONL，或单独一张表）。会话删除只删 UI 可见对话，不动语料。
- **依据**：`direct:` 现 logs 目录删会话即被 `clear_logs_for_session_fs` 删、且 10MB/会话硬顶；eval 明确要 durability。
- **为什么**：数据集价值随时间复利，最怕"用户清会话顺手清空训练集"。生命周期解耦是这需求能不能成立的前提。类比数据库 **WAL（权威、append-only、不删）vs audit log（可丢的派生视图）**——原始 I/O 当 WAL，显示态当派生视图。

### 3. 采集存原文、导出时才脱敏（反转脱敏时机）

- **轴**：A + D　**信心**：中（涉及隐私姿态，需你拍）　**复杂度**：M
- **是什么**：现在是"到达即脱敏+截断"（不可逆，原文永久丢失）。反转成：**本地单用户桌面 app、你就是数据主**，采集时存原文，**脱敏只在导出/分享那一刻**做——且复用项目**已有的 deidentified .db 导出管道**当出口闸门。
- **依据**：`direct:` 两条现有链路都 redact-on-arrival；已存在去标识导出可复用。
- **为什么**：redact-on-arrival 是 eval 数据"没用"的根因之一。本地工具的隐私需求可能根本不需要在采集时脱敏。**这条需要你确认隐私姿态**（采集存原文是否可接受）。

### 4. 每条样本自动带 verification 弱标签 + prompt 版本指纹

- **轴**：A + C　**信心**：高　**复杂度**：S–M
- **是什么**：① 拓扑每次 run 已产出确定性 `verification`（pass/fail），采集时直接当**弱标签**写进样本——零额外标注就有带标签数据集；② 同时记 **SKILL.md 内容 hash + 骨架版本 + model id** 作为分组维度。
- **依据**：`direct:` verification 是现成确定性 gate；项目历史大量工作在改 SKILL.md/prompt。
- **为什么**：标注是 eval 最贵的环节，verification 是在白白浪费的免费标签。版本指纹让你能回答"这次 SKILL.md 改动是进步还是退步"——这才是 eval 真正服务的决策，否则只能看绝对分。

### 5. 退役现有执行日志模块（直接回答你的问题）

- **轴**：E　**信心**：中（取决于你是否还要排障视图）　**复杂度**：S–M
- **是什么**：两个子选项，请你选：
  - **(a) 直接废弃**：eval 采集上线后，删 `diagnostic_store.rs` + `log_file_writer.rs` + `DiagnosticsDrawer` + 三个 Tauri 命令。你已说现日志没用 → 符合"默认单路径、拒绝冗余"。代价：丢失排障时间线可见性。
  - **(b) 降级为派生视图**：保留 DiagnosticsDrawer 外壳，把它的数据源从无用的 milestone 日志切到新 eval store 的视图（list/detail 复用，detail 里加 raw prompt/response）。改动更小，排障入口还在。
- **依据**：`reasoned:` 日志与 eval 在内容/保留/采集点/UI 四处都不重叠 → "合并进同一管道"是最差解；用户称旧数据无用 → 倾向 (a)。
- **为什么**：这是你问题的正面答复——**能合并吗：不该；要不要旧模块：可废弃**。先确认你还需不需要一个排障视图，决定 (a) 还是 (b)。

### 6. 出口：导出 JSONL 数据集 + 当弱模型切换的回放回归门（最高 ROI 的"为什么值得做"）

- **轴**：C　**信心**：中　**复杂度**：M（采集落地后）
- **是什么**：eval store 一键导出 eval-harness 友好的 JSONL（`{prompt, response, toolCalls, label, model, skillHash, runId}`）。更进一步：把历史捕获的真实 prompt 当**重放语料**——切弱模型前用同一批 prompt 跑新模型、按 verification 标签算通过率对比，挂进项目已有的"弱模型 hardening checklist"当**放行门**。
- **依据**：`reasoned:` 项目有明确的弱模型切换计划 + per-run verification；三者一串就是数据驱动的放行门。
- **为什么**：这是整个采集投资回报最高的出口。切弱模型本来要靠手感赌，有真实 prompt + 自动标签就能用数据 gate 决策。也是"为什么现在就要好好存"的最硬理由。借鉴 **Langfuse/Phoenix 的 trace→dataset→eval-run** 三段：导出是"可挑选成 dataset"，不必全量 dump。

## 被否决 / 合并的候选（附理由）

| 候选 | 处理 | 理由 |
|---|---|---|
| 把 eval 字段塞进现有 diagnostic 管道（字面"合并"） | **否决** | 内容/保留/采集点/UI 四处都相反，每处都要开例外——正是这次分析要劝退的方案 |
| 直接拿 `sessions.payload` 当 eval 源导出 | 否决 | 显示态、toolCalls 已截断、随会话删——继承了所有不适合 eval 的属性，零成本陷阱 |
| 在 DiagnosticsDrawer 上补一个导出按钮 | 合并入 #6 | 导出应从原始 store 出，抽屉是给人排障的错位消费方 |
| FDR/CVR 双轨永久共存 | 合并入 #5(b) | 是"降级为派生视图"的另一种说法；但你说日志没用，倾向直接废弃 |
| 回填旧 jsonl 日志做 eval | 否决 | 已截断+脱敏，对 eval 不可恢复；划线"从切换点起干净采集"，零迁移成本 |

## 落地次序（如果你决定做）

1. **先定地基**：#1（worker 边界采原文）+ #2（独立 store）——这俩定了，eval 数据才"存得对、留得住"。
2. **加价值**：#4（verification 标签 + 版本指纹）——低成本，让数据天生 eval-ready。
3. **定隐私姿态**：#3（采集存原文 vs 采集即脱敏）——**需你拍**。
4. **退旧**：#5 选 (a) 废弃还是 (b) 降级——**需你拍**。
5. **出口**：#6（JSONL 导出 + 回放门）——采集稳定后做。

## 已定决策（2026-06-25 boss 拍板）

- **#3 隐私姿态**：eval 采集**存原文，导出也不脱敏**；脱敏只保留在 **agent↔用户交互（聊天展示）** 那一层。即：worker 边界采原文 → eval store 原样落 + 原样导出；现有 redact 管道只服务聊天显示态。
- **#5 旧模块**：**直接废弃**执行日志模块（删 `diagnostic_store.rs` / `log_file_writer.rs` / `DiagnosticsDrawer` + 三个 Tauri 命令）。
- **下一步**：用 `ce-brainstorm` 把 eval 采集（#1+#2+#4+#6 + 废弃旧模块）细化成 requirements。
