Ideation: 场景体系 + 5 跳线性拓扑 + skill 按场景拆分

date 2026-06-11 · topic scenario-skill-split · mode repo-grounded
focus 支持规范图 5-1 的 5 跳线性拓扑（宇航场景）；保持通用 + 宇航两场景、场景下多拓扑、预留工业； topology skill 拆为主 SKILL.md 索引 + per-scenario reference；移除 skill 内 package.json

两个改变全局的核查发现（3 个独立子代理交叉核实）：
① 前提校正——「worker 不知场景、按场景注入不可行」是错的：scenarioConfigId 已经经 agent-adapter.ts:160 → commands.rs payload → claude-agent-worker.mjs:94 送达 buildSystemPromptForStage 第一参数，只是当前以 _stageRunnerInput 弃用。按场景确定性注入一行可达。
② 发布阻断——播种是目录粒度「已存在即跳过」（skill_files.rs:331）：本次拆分是第一个「向已播种目录新增文件」的特性， 不升级播种粒度，所有存量 release 安装升级后拿不到 references/ 与新索引——拆分对老用户静默不生效。

Grounding Context

  • 场景现状：src/domain/scenario-config.ts 已有 generic-tsn（通用）+ aerospace-onboard（宇航）； 消费方为 UI 文案与 stageRunnerInput.scenarioConfigId。其 defaults.topology 为零消费死配置， 且与 SKILL.md 推荐默认已漂移（endSystemsPerSwitch 5 vs 2）。
  • 模板 catalog（topology_compute.rs:92-196）：generic-line / generic-ring / dual-plane-redundant（已实现）。 模板与场景无机器可读关联——靠 87 行 SKILL.md 决策树散文，且决策树按用户措辞分支、从不引用场景 id。
  • 注入链路：worker 整文件读 skillRoot/tsn-topology/SKILL.md 拼单字符串注入（sentinel 分隔； string[] 会崩 redactSecrets）；agent 工具白名单含 Read + 8 个 MCP 工具；skillRoot 经 payload 同源（刚 merge）。
  • 5 跳线性（规范图 5-1 + 5SW_2NIC 配置）：E1→SW1→…→SW5→E2，端系统仅两端，S 形折叠两行； generic-line 表达不了「仅两端挂 ES」。
  • 官方规范背书：SKILL.md + references/ 是 Anthropic Agent Skills 官方 Pattern 2（bigquery-skill 同构先例）； SKILL.md 建议 <500 行；references 一层深；package.json 不在规范内（残留，删除正确）。
  • 既有决策张力：2026-06-09 ideation Rejection #4（不拆 reference）被本次显式反转（论据：官方 Pattern 2 + 场景扩展预期）； Rejection #1（能参数化就不建新模板）在 5 跳线性裁决中得到遵循。

置信 90% · 复杂度 Medium1. 播种升级为文件粒度三态判定（出厂 manifest + 哈希）——拆分的发布前置 D

Description
播种时写出厂 manifest（文件 → 出厂内容哈希）；此后每次解析逐文件三态判定：app-data 缺失 → 补播； 存在且哈希 == 出厂值（用户没改）→ 随新版本静默更新；用户改过 → 保留不动。 deferred 的「重置为内置版本」按钮退化为「按 manifest 恢复」的免费副产品。
Basis
direct: skill_files.rs:331「目录已播种即跳过」——存量安装升级永远拿不到本次新增的 references/*.md 与新版索引 SKILL.md； external: dpkg conffile md5sums 数据库 + RPM .rpmnew 的成熟三态语义（包升级 = app 升级、conffile = skill 文件、md5sums = manifest）。
Rationale
七个子代理 idea 收敛到同一结论：不解决播种粒度，拆分 + 5 跳指引 + 删 package.json 对所有老用户全部不生效且无报错 （worker 读到旧单体 SKILL.md 是合法路径）。这是本批唯一「不做等于没发布」的链路项。manifest 方案同时解决「新增文件送达」与 「未编辑文件的内容升级」（文件粒度补缺只解决前者——存量用户的 SKILL.md 本身要从 87 行单体换形为索引）。
Downsides
manifest 自身是新状态文件（损坏/缺失需兜底语义：缺 manifest 视同全部「用户改过」保守不动）； 比纯文件补缺多一层哈希对账逻辑。被拒的替代：overlay/COW 双层文件系统更优雅但推翻刚 merge 的播种架构、读路径双端全改。
dpkg conffile 式三态：新文件可送达、改进可送达、用户编辑不丢

置信 85% · 复杂度 Medium2. worker 按场景确定性注入：主索引 + 当前场景 reference + 绝对路径表 C

Description
worker 把目前弃用的 _stageRunnerInput 用起来：按 scenarioConfigId 拼接「主 SKILL.md 索引 + 当前场景 reference」 单字符串注入（仍走 sentinel 契约）；未知/缺失场景沿 fallback 链回退通用（i18n locale 回退语义，给工业预留现成机制）。 注入正文末尾由 worker 运行时追加「可用参考文件」绝对路径表（枚举 skillRoot/tsn-topology/references/*.md）， agent 会话中途需要其他场景时可直接 Read——路径具体到可执行，不靠索引里的相对路径祈祷。
Basis
direct: agent-adapter.ts:160 → worker:94/376 链路核实（场景 id 已送达、形参弃用）； claude-agent-worker.mjs:363 allowedTools 已含 Read；release 下 agent 自行猜路径会读到 Resource 工厂副本 （与刚修的「反向不同源」bb1e59b 同病）。
Rationale
「全量拼接（为不在场的场景付 token）vs 索引 + 赌 agent 自觉 Read（可靠性）」的两难因前提校正直接消失： 活跃场景指引确定在场、非活跃场景零 token、注入体积与场景数解耦。绝对路径表堵死拆分后 PR#21 休眠病的复发条件 （references 默认不在注入路径上）。
Downsides
worker 增加一次目录枚举与场景文件选择逻辑（含 fail-open：场景文件读不到 → 注入索引 + 审计警告）； 会话中途切场景依赖 agent 主动 Read（小概率路径，可接受）。
场景 push 为主、绝对路径 Read 为兜底；工业缺文件沿链回退通用

置信 85% · 复杂度 Small-Medium3. 内容切分三层判据：协议不变量上移骨架，索引装「背的」，reference 装「翻的」 C

Description
拆分时按内容性质三层归位，判据各一句话：协议不变量（「不要写 stage-result」「重试逐字节复用」——用户改坏会破坏 worker 对账） 上移进 SYSTEM_PROMPT_SKELETON 代码常量，脱离可编辑面；主 SKILL.md = agent 调第一个工具之前就必须知道的 （领域语义表、回复边界、场景路由表）；per-scenario reference = 确定场景之后才消费的（该场景模板选择、推荐参数、规范图 preset）。 配套一行 UI 标注：Skill 面板每个文件注明消费方式（「每次运行注入」/「按场景注入」/「agent 按需读取」）。
Basis
external: 航空 QRH/FCOM 双级体系——memory item（闭卷背诵）vs read-do checklist（翻开照做）的切分准则； direct: 现 SKILL.md 87 行中「当前边界/回复边界」（15-22 行等）与「场景决策树/推荐默认」（41-68 行）性质迥异同权限混装， 骨架常量与 SKILL.md 已存在两处维护同类禁令的漂移。
Rationale
官方 Pattern 2 只给文件结构、没给内容归属判据——没有判据，拆分时每段文字去哪会变成无尽 review 扯皮； 且只按场景拆会把「用户一改就崩」的协议复制进 3 份 reference，错误面 ×3。这条判据让「可编辑指引事实源」的承诺只覆盖改了也安全的内容。
Downsides
骨架常量扩大意味着这部分指引不再可热编辑（这恰是目的，但要在迁移时逐条确认哪些真是协议、哪些其实是可调指引——错划会把本想让用户调的东西锁死）。
QRH 准则：背的进脑子（骨架/索引），翻的留手册（reference）

置信 80% · 复杂度 Medium4. 场景→模板映射机器化：descriptor scenarios 字段 + reference 内命名 preset 表 + 三方对账 lint A

Description
三件套：① catalog 每个模板 descriptor 加 scenarios 数组（复用既有 tags 机制，多对多天然支持——generic-line 同时属两场景）； ② per-scenario reference 的主体是命名 preset 表：「规范图 5-1 / 5SW_2NIC」→ {templateId + 完整参数 JSON + 布局说明}——用户说「按图 5-1 来」 agent 查行照填，规范符合从概率行为变查表；③ verify-skills 加三方对账 lint：reference 引用的 templateId 必须在 catalog、 声明了场景的模板必须被该场景 reference 承载、场景 reference 文件名 == ScenarioConfigId（工业预留 = 这条命名契约本身，本期零占位文件）。
Basis
direct: descriptor 已有 tags 数组（topology_compute.rs:109/125/139）但全链路零消费；SKILL.md:12 已确立 「合法域以 describe_templates 为准」的单源先例——场景适配是同一模式延伸；external: Cisco Validated Designs （行业指南 = 场景 reference、公共构件库 = 模板 catalog、named design = preset 行）。
Rationale
映射留在散文里，每加模板/场景就多一处人肉同步且无校验；机器化后「工业场景上线」= 写一个 reference 文件 + 模板打标签， agent 一次 describe_templates 同时拿到合法域与场景适配。preset 表同时是「场景下多种拓扑结构」的可枚举落点。
Downsides
preset 放 reference（可编辑、文档化）意味着用户可改坏 preset 参数——由 catalog 参数校验兜底（错参 → requires_clarification，不会静默生成错拓扑）； scenarios 字段与 reference 的双向覆盖断言需要 lint 解析 markdown，verify-skills 复杂度上升一档。
结构归 catalog、场景语义归 reference，漂移由闸抓而非人肉

置信 85% · 复杂度 Medium5. 5 跳线性 = generic-line + endSystemPlacement 参数 + boustrophedon 蛇形布局 B

Description
不开新模板（遵循 Rejection #1）：generic-line 加可选参数 endSystemPlacement: "per-switch" | "ends-only" （缺省 per-switch 全向后兼容），ends-only 时 ES 仅挂链两端。布局不加旋钮：线型族按 switchCount 与画布纵横比的确定性公式 触发蛇形折叠（boustrophedon——3D 打印 infill / 扫地机覆盖路径的标准算法：固定宽度内回折、行向交替、U 弯换行）， 5 跳折成两行正好对齐规范图 5-1，12 跳自动多折几行。测试关卡先行：5SW_2NIC 确定性坐标快照 + 折返链路 slab 防穿框断言 + 「对图不对文」校验（直接对照转出的 5-1 png）。宇航归属经 idea 4 的 preset 表达（模板保持场景中立）。
Basis
direct: generic_distributed_params（topology_compute.rs:198-208）仅三个均匀参数、表达不了仅两端挂载； END_SYSTEMS_PER_SWITCH_MIN=1 硬编码均匀挂载假设；external: boustrophedon 覆盖路径分解（确定性蛇形算法，任意长度自适应）。
Rationale
参数化让 line 族复用既有八关卡管线（descriptor→zod→validate→排序规范→确定性测试→validate_intermediate→SKILL→drift 守护）的已付成本； 蛇形公式一次解决 5 跳和未来任意长链，不为一张图硬编码坐标。被拒的替代：挂载向量新模板（PROFINET 式）与 explicit 模板更通用但 YAGNI—— 显式列表风格已有 dual-plane 承载，真需求出现再升级。
Downsides
generic-line 生成器内出现布局分支（直线 vs 蛇形），确定性测试矩阵增加一档；ends-only 打破「每台 SW 接入数相同」的端口定偏移前提， 端口分配需改 first-free 游标（dual-plane 已有先例）。折返处节点距离近，标签防撞需复用按侧分层机制。
规范图 5-1 的 S 形即 boustrophedon 公式在 switchCount=5 的取值

置信 80% · 复杂度 Small-Medium6. 打包链路换整目录映射 + skill 目录纯文本白名单契约（含删 package.json 与残留清理） D

Description
tauri.conf.json 资源从逐文件映射改为 skills 整目录映射（新增 reference 零 conf 改动）；verify-skills 职责反转： 「逐文件断言映射」→「目录映射存在 + 文件 git 跟踪 + 正向白名单（skill 目录只允许 SKILL.md 与 references/*.md 文本文件）」； commands.rs:805 的逐文件点名断言（package.json 正向断言 + 5 条 legacy 黑名单）整体替换为结构性断言（目录映射存在 + 磁盘无 package.json）。 顺带清理同类残留：scenario-config.ts 的 defaults.topology 死配置（零消费且已与 SKILL.md 漂移：endSystemsPerSwitch 5 vs 2）。
Basis
direct: tauri.conf.json:36-41 逐文件映射（:37 即待删的 package.json 行）；verify-skills.mjs:41-52 逐文件断言； grep 验证 defaults.topology 零消费方、与 SKILL.md:46 漂移；external: Tauri 2 bundle resources 支持目录映射 （落地前需 5 分钟核实尾斜杠/递归语义——本 idea 唯一外部不确定点）。
Rationale
本次要删 1 个文件 + 加约 3 个文件，是「每加一个文件交三处同步税」的第一现场；目录映射 + 白名单把整类问题 （漏映射、悬空映射、误放脚本、二进制混入）一次吃掉，未来 Phase B 各阶段 skill 目录自动继承。逐文件黑名单随历史线性增长且只防已知病。
Downsides
白名单收紧后未来想放合法新类型文件（如 examples/）需改闸规则一次；目录映射语义若与预期不符（不递归等）需回退逐文件并接受同步税。

置信 75% · 复杂度 Medium7. 规范图配置目录 → 表驱动黄金测试管线 B

Description
把刚入库的 docs/prototypes/典型组网测试配置/（2SW_6NIC / 4SW_4NIC / 5SW_2NIC）变成生成器确定性测试的 fixture 源： 每张规范图一行表驱动数据（templateId + 参数 + 期望节点数/邻接关系/ES 挂载位置），断言模板输出与规范拓扑同构。 以后每对标一张新规范图 = 配置进目录 + 表加一行。
Basis
direct: 三套规范配置已入库（commit 64e0af2）；topology_compute.rs 测试区已有布局断言先例 （行内间距断言、slab 相交检测、dual_plane spec projection 测试）。
Rationale
「画布对标规范图」目前靠真机肉眼验收（本周成列布局 bug 正是真机才被抓出）；fixture 管线把规范符合性变成回归资产—— 改布局常量时 CI 自动守住已验收图形，工业规范图进来时验收成本近乎为零。
Downsides
规范 json 的 sync_type 码表跨配置不一致（4SW_4NIC 的 type=4 是 SW、5SW_2NIC 的 type=2 是 SW），fixture 解析层要做适配映射； 同构断言（邻接而非坐标逐点比对）需要一次性设计，避免把手拖过的自由坐标误当规范。

Rejection Summary

被拒/合并 idea原因
overlay/COW 双层文件系统取代播种（2 个变体）被 #1 manifest 三态替代——更优雅但推翻刚 merge 的播种架构，编辑器/worker 读路径双端全改，性价比低
纯文件粒度补缺播种（3 个变体）并入 #1 为其子集——只解决「新文件送达」，不解决「未编辑文件的内容升级」（SKILL.md 本身要换形为索引）
纯 pull 注入（索引 + 全靠 agent Read）赌 agent 自觉，多一次工具往返且 Read 失败静默降级不可观测；其绝对路径表部分并入 #2
全量拼接所有 references为不在场的场景付 token 且场景间互相干扰；260 行尚可但随 Phase B 各阶段指引线性膨胀
preset 进 Rust catalog（防误编辑）与 boss「场景文档承载」意图相悖；可编辑性即拆分的目的之一，防错由参数校验兜底（#4 采 reference 方案）
ScenarioConfig 注册表挂 templateIds与 descriptor scenarios 字段构成双源——选 catalog 单源（#4），scenario-config 维持 UI 配置职责
reference frontmatter 即场景注册表对账机制并入 #4 的 lint；frontmatter 作注册表会与 scenario-config.ts 形成第二事实源
多对多「适用关系」独立关系表被 #4 preset 表实质吸收（preset 行即关系行）
linear-chain 挂载向量新模板（PROFINET 式）ends-only 枚举已覆盖图 5-1；向量形态 YAGNI，真需求（工业 PROFINET 任意挂载）出现再升级参数形态
explicit/custom 显式列表通用模板显式列表风格已有 dual-plane 承载；为「任意规范图」泛化是 YAGNI，且把布局责任推给 agent 反而降质
endSystemPlacement 的 per-switch map 形态超出图 5-1 需求；二值枚举够用且保留升级空间（map 是其超集，向后兼容可加）
「布局制度数」模板判据 / S 形几何雷区清单 / 参数化吃三图裁决性内容与测试关卡并入 #5 的 rationale 与验收口径
i18n fallback 链 / QRH 切分判据 / 可编辑性边界 / 运行时路径清单（5 个变体）机制与判据分别并入 #2（fallback、路径表）与 #3（内容三层归位）
describe_templates scenarios 过滤规则 / manifest 审核双向断言 / 场景路由键（4 个变体）并入 #4 三件套（字段 + lint + 路由经 #2 注入）
幽灵 SKILL_IDS 清理保留——id-only 预留是仓库既有预留模式（tsn-time-sync/tsn-inet-export 即 Phase B 占位），与工业场景命名契约同构
Skill 面板消费方式标注缩为 #3 的配套一行（独立成项不过会议门槛）
defaults.topology 升格为机器可读默认源反向裁决：它是零消费死配置且形状只能表达 generic 族（dual-plane/5跳塞不进），删除并入 #6；默认值事实源维持 SKILL/reference
