# 拓扑 skill「确认/澄清不足」问题分析（ce-ideate 种子）

boss 的核心反馈：**(1) 没跟用户确认场景就直接出拓扑图；(2) 与用户的确认内容过少。** 主要改进方向是 **skill 层**。下面把 Excel「测试结果/备注」列的实测观察 + 「体验性问题」sheet 归纳成主题，每条挂具体用例。

## L1 硬不变量（文档自定，作为底线）
链路两端有效；端口不重复占用；无自环/重复链路；**不静默破坏**；查询不改状态；change card 忠实。

---

## 主题 1：场景没确认就套固化范式生成拓扑（boss 焦点 #1）
- **A-C-02**（创建箭载 TSN 拓扑，规模缺省）：实测「有追问、有推荐，但图4-1 描述莫名其妙；**过于依赖 skill 中固化的范式**，建议以追问节点数量、特征、类型等进一步明确需求」。
- **G-C-07**（建个环网，缺规模）：实测「缺少关键信息时仍然可以创建拓扑；缺必填追问——这是否符合设计原则」。
- 根因假设：SKILL.md 初始化路径偏向「直接套模板默认蓝图生成」，缺一道**强制的场景画像/规模/特征确认**环节；箭载场景尤其把固化范式当默认。

## 主题 2：该追问时不追问 —— 与用户确认过少（boss 焦点 #2）
- **指代歧义直接动手**：G-D-03（一端连两交换机，删线时 agent 直接删 1 条、缺追问）；G-U-04（挪线同样歧义）。
- **引用不存在不追问、直接造**：A-U-01（要把双归属改到 SW1/SW3，无 SW3，agent 没澄清直接新建 SW3 并改连接；命名异常）。
- **目标模糊不澄清**：A-U-06（「让拓扑更高可靠一点」应澄清双/三冗余等候选）；A-U-04（双冗余升三冗余默认新建节点+连线，未问是否新增第三交换机）。
- **缺连接/孤立节点的处理与设计相反**：G-C-03（设计期望「先建孤立节点再建议连接、不阻断」，实测「一定会自动连接、不会留孤立、缺信息也不追问」）。
- 根因假设：SKILL.md「已有拓扑编辑路径」虽写了「指代不唯一先澄清」，但实际 agent 倾向直接执行；缺**强制硬规则**：歧义 / 缺必填 / 引用不存在 → 必须先澄清，不得擅自补全或新建。

## 主题 3：破坏性操作只「告知」、不「卡确认」
- **A-D-02**（删 E1 到 SW2 的链路，破坏双归属）：期望「未确认不得删除」，实测「会执行 + 明确告知失去冗余」。
- **A-D-03**（删 SW1，破坏整体冗余）：期望「未确认不得执行」，实测「会执行 + 告知降级为单路径」。
- 根因假设：破坏冗余/结构的操作走的是**告知式**（先做再说），而非**阻断式确认**（先确认再做）。注意这与「确认并继续」是两件事——这里要的是**操作级**的破坏性确认。

## 主题 4：命名规则混乱 / 不一致
- **A-C-03**（新增 E5）：实测「描述新增 e5，但创建的是 ES-6，且与其它端节点命名不一致」。
- **A-U-01**：命名异常。
- 体验性问题：「SW 有大写有小写，端节点命名在不同工程中不一致」「怀疑节点名称与实际节点名称不一致」。
- 根因假设：用户叫法 / 显示名 / 库内 sync_name 三者映射规则没在 skill+MCP 层固化统一，新增对象的命名/编号分配不可预测。

## 主题 5：模型能力 vs 用户预期错位（非确认问题，但影响体验）
- **端口数量不是可存储属性**：G-U-01（E1 改 8 口）/G-U-06（SW1 缩 1 口）/G-Q-04（还能再接几个）——模型按「实际接线数」算有效端口，不存总口数；用户用「口数」表达的需求无法直接落地。boss 备注「是否符合甲方预期」。
- **不支持重命名**：G-U-02。
- 方向：skill 需要识别这类「无法直接表达」的需求，**解释能力边界 + 引导到可表达的等价操作**（如改连接关系），而不是静默失败或乱改。

## 主题 6：交互/流程体验（体验性问题 sheet）
- **「确认并继续」语义歧义**：容易被理解为「确认当前这个小任务」，而非「结束当前大阶段、进入时间同步/流量」。（与本周期刚做的确认流程直接相关。）
- 思考态：无明显状态、长任务莫名等待、不支持中断、思考中发送按钮未置灰。
- **撤销/回退兜底**：用户用自然语言描述回退，但「自然语言回退」与「代码级回退」结果不一致，缺兜底与回退限制说明。
- 时间同步后无法改拓扑——是设计还是没闭环（注：本周期 PR#26/#34 做过切阶段回退，可能未闭环好）。
- 双平面是否代表隔离（语义待明确）。

## 给 ideate 的方向锚（skill 层为主）
1. **场景确认前置**：初始化/重大生成前，强制确认场景画像 + 规模 + 关键特征，而非直接套固化模板。
2. **澄清硬规则**：歧义 / 缺必填 / 引用不存在 → 必须澄清，禁止擅自补全或新建对象。
3. **破坏性操作阻断式确认**：破坏冗余/结构的操作「先确认再执行」，区别于「确认并继续」的阶段推进。
4. **命名/编号一致性**：用户叫法↔显示名↔sync_name 映射 + 新对象命名规则在 skill+MCP 固化。
5. **能力边界引导**：识别「口数/重命名」等模型不直接支持的需求，解释 + 引导等价操作。
6. **交互澄清的呈现**：澄清用中文编号选项；「确认并继续」语义消歧；回退兜底。
