# Ideation：拓扑 skill 重写、prompt 统一与确认/澄清

起因是 boss 的两点反馈：没确认场景就出图、跟用户确认太少。几轮下来方向扩成了：进门先定场景、整份 skill 重写、validate 做成用户能问的可行性检查、模板按 id 取、命名统一、把散在代码里的 prompt 一起理清、再加一项推理态的交互改造。问题分析在 `docs/ideation/2026-06-18-topology-confirmation-skill-analysis.md`。

## Grounding Context

### agent 的 prompt 现在是怎么拼的

系统 prompt 三段拼接（`src-node/claude-agent-worker.mjs`）：

1. `SYSTEM_PROMPT_SKELETON`（worker.mjs:451）——一段约 700 字的中文硬骨架，放最前，写死了角色、阶段顺序、「前进只走确认并继续按钮」、`request_stage_change` 语义、重试规则等。注释说明这是协议不变量，用户不该改，所以没进 SKILL.md。
2. `<<<SKILL_GUIDANCE>>>` 注入 `SKILL.md` 全文（可编辑事实源）。
3. `<<<SCENARIO_REFERENCE>>>` 注入 `references/<场景id>.md`（按 `scenarioConfigId` 选）。

但除了系统 prompt，`buildPrompt`（worker.mjs:563-622）还在**用户轮**里塞了一大堆硬文案：结构化结果规则、执行顺序、交互规则 8 条、回复要求 7 条……这些跟系统 prompt 的职责是重叠的。再加上 MCP 工具的 description 里也夹着大段中文行为指令（`inspect` 那段超长、`validate` 的「必须把错误转告用户 / initialize 后别马上 validate」）。

结果就是**同一条规则散在多层、措辞还各不一样**。盘点出来明显该统一的：

| 规则 | 散在哪 | 问题 |
|---|---|---|
| 重试逐字节复用同一 batch | 骨架、buildPrompt、apply_operations description | 3 处措辞不同；**骨架用的是废弃键名 `imac`，description 用 `syncName`，直接打架** |
| 节点身份键名 | 骨架(2 处 `imac`)、SKILL/description(`syncName`) | PR #35 已统一成 `sync_name`，骨架没跟着改，残留 `imac` |
| initialize 后不复检 validate | 骨架、SKILL.md、validate description | 一句话写了 3 层 |
| 仿真/OMNeT++ 没接入 | 骨架、buildPrompt、会话上下文注入、硬替换兜底（共 4 处） | 4 处措辞不一，其中两处几乎一字不差纯重复 |
| 显示名映射规则 | SKILL.md、inspect description | 两处覆盖同一规则，inspect 里还多了 plane 颜色语义、SKILL 里没有 |
| 「切回阶段」确认文案 | request_stage_change description、agent-adapter event、chat-pane UI（3 层） | 各写一遍，改措辞要改 3 处 |
| verify 错误文案 | Rust `message_zh`、TS `composeVerificationBlockText` | Rust 和 TS 两层各一份，几乎重复 |

### 命名现状

- 初始化时名字硬编码在 `src-tauri/src/topology_compute.rs`：交换机 `SW-{n}`、端系统 `ES-{n}`（已经是 SW/ES 开头，但**只在 initialize 这条路**）。
- `node_add` 操作没有 name 字段（`topology_ops.rs` L23-31），agent 后续新增的节点名字是空的、序号还是自己随手挑的——所以会出现「说 e5、建成 ES-6」。
- validate（`src-tauri/src/topology_verify.rs`）现在校验：重号、类型未知、链路重号、连到不存在的节点、缺端口、孤立节点、无端系统、不连通——**没有命名前缀校验**。

### 推理态 UI 现状

- 全局 loading 是 `AgentRunStatusBar`（chat-pane/index.tsx:219），渲染在 `App.tsx:386`，夹在 header 和 main 之间，一条顶部全宽 banner，订阅 `isAgentRunning` + `agentRunPhase`（idle/connecting/streaming/waiting，纯 useState、无全局 store）。
- 输入框 textarea 推理中**不禁用**、可以继续打字，但 Enter 在推理中静默不发送；占位文案不变。
- 发送按钮推理中 `disabled=true`、永远是纸飞机图标，没有「终止」形态。
- **终止能力完全没有**：前端没有任何 stop/cancel 入口，后端 `run_claude_agent` 是 `spawn_blocking` + `CLAUDE_AGENT_RUNNING` 单例锁（commands.rs），没有取消通道，只有超时才 kill 进程组。要做终止得新建三段链路：后端新 Tauri 命令（向 worker 进程发信号 + 清 flag）、前端 `stopRun` action、按钮改 Stop 形态。

### 几条不能碰坏的底线

- **分层**：会把数据搞乱的硬规则在 worker 骨架，措辞和默认值在 SKILL.md，参数合法范围在 MCP（Rust 单一来源）。
- **`AskUserQuestion` 工具被全局禁用**（worker 里 disallowedTools）——所以「澄清给编号选项」只能让 agent 在对话里输出文字编号，不能指望那个工具。
- 「确认并继续」按钮保留确定性推进（不走大模型）。安全判断不靠 prompt。
- 改完 worker 的 prompt 要 `build:worker`（worker 跑 dist 产物）；指引注入用字符串拼接、不能传数组。

## 涉及哪几块

- prompt 体系：散落的指令/文案怎么归位、去重、统一口径
- 场景：进门怎么定、定了算不算会话状态
- skill 本身：怎么重写、规则放哪层
- 澄清与确认：什么时候停下问、什么操作先打招呼
- 命名与能力边界：SW/ES 统一、做不到的怎么回应
- MCP 能力：validate 当可行性检查、describe_templates 按 id 取
- 交互：推理态的 loading、输入框、终止

## Ranked Ideas

### 1. 先把 agent 的 prompt 体系理清楚，再谈重写 skill

**这是什么：** boss 说得对——代码里的 prompt 太散，agent 行为就飘。重写不能只动 SKILL.md，得把整套面向 agent 的文字一起规划：先定清楚「每一类指令该住在哪一层」（① 协议不变量 → worker 骨架；② 领域指引和措辞 → SKILL.md / reference；③ 单个工具怎么用 → 该工具的 description），然后按这个归属把现在散在骨架、`buildPrompt`、会话上下文注入、MCP description 里的东西各归各位，**同一条规则只留一处**。`buildPrompt` 那一大坨用户轮硬文案要么并进系统 prompt、要么并进 SKILL，别再自成一摊。顺手修掉盘点出的硬伤：骨架里残留的 `imac` 旧键名、4 处不一致的「仿真没接入」、3 处「切回阶段」文案。

**属于：** prompt 体系（也是 skill 重写的前提）

**依据（看代码）：** 上面 grounding 那张表——7 类规则各自散在 2-4 处、措辞不一，骨架 `imac` 跟 description `syncName` 直接矛盾。系统 prompt 三段拼接之外 `buildPrompt` 还另塞散文案（worker.mjs:563-622）。

**为什么值得做：** 这是 boss 这轮的核心判断，也是其它点子能稳的地基——规则只有一处、口径一致，agent 才不会被自相矛盾的指令带偏。重写 skill 如果不先把「谁该住哪层」定下来，只是把乱摊子从一个文件挪到另一个文件。

**代价 / 要注意：** 这是最大的一块，得先有一张「指令清单 + 归属表」才能动手，适合下一步用 ce-brainstorm 专门拆。动 worker 骨架风险高（协议不变量，写错会破坏对账），每改一处要想清楚分层、改完 `build:worker`。范围跨 .mjs / .ts / .rs / .md 多个文件，要一次盘清不然容易漏。

**把握：** 中（方向明确，但工程量大、要专门拆） · **改动：** 大

### 2. 进门第一句话先让用户定场景，定了当成会话的固有状态

**这是什么：** agent 一打开，先问「你这是哪类网络？」给明确选项（航空箭载 / 通用工业 / …）。用户选定后，这个场景就是整个会话的底色，后面所有生成、编辑、选模板都默认在这个场景下，不主动说「换场景」就不变。航空箭载的双平面冗余和工业本来就不一样，场景定错后面全错。**这版先不做「中途切换场景」，但流程和数据上要留出位置**：场景是个能读、也能被显式改写的会话状态，而不是写死在会话创建时的一个 id。

**属于：** 场景

**依据（看代码 + 实测）：** `scenarioConfigId` 会话创建时注入、未知静默回退 generic-tsn（SKILL.md L19/L25），agent 拿到就套模板、不跟用户对。实测 A-C-02 缺规模也照生成。

**为什么值得做：** 场景是后面所有判断的根，先用一句话定下来比生成完返工划算。「场景当会话状态」现在不留口子，以后想做切换会很难补。

**代价 / 要注意：** 多一步开场交互，给个「先按通用来」兜底就行。`AskUserQuestion` 被禁用，所以这个「选场景」要么用对话里的文字编号选项、要么做成前端的一个选择控件（不走大模型），后者更稳。中途切换这版不做，但别堵死路。

**把握：** 高 · **改动：** 中

### 3. 让 validate 变成用户随时能问的「这么改行不行」

**这是什么：** 现在 validate 是「apply 之后调一次确认结构没坏」。但用户常常是动手前先问：「我把这两台连起来行吗？会不会弄出孤立节点？」——这是改动前的可行性咨询。validate 本来就会算连通性、孤立节点、转发可达，应该让它能对一个**还没落库的假设改动**先算一遍、把结论用人话讲给用户，用户说行了再真动手。现在「传草稿 JSON」那条路只做 schema 校验、不做这套结构检查，这是个实打实的缺口。

**属于：** 确认 + 能力边界

**依据（看代码）：** `topology.validate` 无参查已落库拓扑的完整结构、传草稿只做 schema 级（topology-tools.ts L86-95），所以「对没落库的改动做完整结构预检」目前做不到。

**为什么值得做：** 用户问「能不能这么改」时要的是「会不会出问题」，不是 agent 直接替他改。validate 能预检，agent 就能先答「这样会让 SW2 变孤立，要不要换个连法」，也给点子 6（破坏性确认）兜底——判断由确定性检查给，不靠 agent 嘴上保证。

**代价 / 要注意：** 要让 validate 支持「对假设拓扑做完整结构检查」是 MCP/Rust 改动，**得你点头**。还要想清楚「假设的改动」怎么表达——在当前拓扑上叠一批待定 operations，还是传完整草稿。

**把握：** 中 · **改动：** 中

### 4. 模板别一次全返回，按 id 或名字取；常用 id 写进 skill

**这是什么：** 现在 describe_templates 要么按场景过滤、要么全吐出来，每个还带完整参数。模板少时无所谓，将来十几个会占上下文、让 agent 挑花眼。改成默认只返回 id + 名字 + 一句话说明的清单，agent 挑中再按 id 取那一个的完整参数。常用模板的 id 直接写进 SKILL，让 agent 一眼知道「箭载图 4-1」对应哪个 id。

**属于：** MCP 能力

**依据（看代码）：** describe_templates 传 scenario 过滤、不传返回全量目录、带完整参数（topology-tools.ts L36-46）。

**为什么值得做：** 这是会随模板变多而变糟的问题，早点把「列清单 → 按 id 取详情」拆成两步，以后加模板不撑爆上下文。id 进 skill 也让「用户说图 4-1」到「该用哪个模板」更稳。

**代价 / 要注意：** 要给 describe_templates 加「只列清单」和「按 id 取一个」两种用法（MCP 改动，**得你点头**）。SKILL 里写的 id 要跟 Rust 的模板源对得上。模板还少时，这件事排在 1-3 后面。

**把握：** 中高 · **改动：** 中

### 5. 拿不准就开口：该问的问、做不到的直说

**这是什么：** 现在 agent 太爱自己脑补。该停下来跟用户说话的情况，归一类处理，别擅自动手：

- 一个叫法对上好几个节点 → 列编号让用户选（这条现在有）。
- 让连到 SW3、可库里没 SW3 → 说「没有 SW3，你是要新建还是指现有的某台？」，别默默建一个。
- 缺关键信息、或目标很虚（「更可靠点」）→ 摊成具体选项问（「双冗余还是三冗余？」），别自己拍板加节点。
- 根本做不到的（把 E1 改成 8 口、给节点改名）→ 别假装改了也别乱改，直说「口数是按实际接线算的，你是想多接几条线吗」，给个能落地的替代。

**属于：** 澄清 + 能力边界

**依据（看代码 + 实测）：** SKILL.md L61 只写了「引用不唯一」要问，没覆盖不存在/缺信息/目标模糊。实测 A-U-01 引用不存在的 SW3 直接新建还命名乱了、A-U-06「更可靠」没问就加节点、G-C-07 缺必填也照建；G-U-01/G-U-02 口数和重命名落不了地。

**为什么值得做：** 这是 boss 说「跟用户确认太少」最直接的一块。脑补错一步后面全歪，还连累命名。做不到的说清楚，是 boss 问的「符不符合甲方预期」——甲方可能不懂 TSN，用「口数」表达需求，agent 得听懂并引导。

**代价 / 要注意：** 拿捏分寸——明显的别问、不确定的才问、一次别问太多。「该不该停下问」放 skill 措辞容易被忽略，有一部分得进硬规则（比如「引用不存在的对象禁止直接新建」），这是重写时（点子 1）要定的分层。编号选项只能用对话文字，`AskUserQuestion` 用不了。

**把握：** 高 · **改动：** 中

### 6. 删之前、改坏冗余之前，先打招呼再动手

**这是什么：** 用户说「删掉 E1 到 SW2 那条线」「删了 SW1」，如果这会把双归属打成单点、或把整体冗余降级，现在 agent 是删完了顺便告诉你「已降级」。应该反过来：先说「这条删了你就失去 SW1 的冗余、降成单路径了，确定删？」，用户点头再删。注意这跟阶段之间的「确认并继续」是两码事——那个是「这一阶段做完进下一阶段」，这个是「这个具体操作有破坏性、先拦一下」，别混、别共用一个通道。

**属于：** 确认

**依据（看代码 + 实测）：** 删除现在只在 Rust 端报 `NodeHasLinks`（topology_ops.rs L197），没有「先讲后果、等确认」这层。实测 A-D-02/A-D-03 都是先删了再告知降级。阶段级「确认并继续」是另一套（确定性推进、不走大模型），boss 已定调保留。

**为什么值得做：** 删是不可逆的，「先做再说」和「先问再做」差别很大。「这步算不算破坏性」最好交给点子 3 的 validate 预检判断，不靠 agent 自己掂量。

**代价 / 要注意：** 「破坏性确认」怎么跟现在「打字走大模型、按钮才确定性推进」这套接上要设计清楚，别搞乱已定的按钮行为。先确认再做会让删东西多一步，但这步是该有的。

**把握：** 中高 · **改动：** 中

### 7. 命名统一：交换机一律 SW 开头、端系统一律 ES 开头，写进 skill 也写进 validate

**这是什么：** 把命名规则钉死成一条：交换机 `SW` 开头、端系统 `ES` 开头，序号按固定规则来。两头都管：

- **skill 层**：SKILL 里写清这条命名规则，新增对象时 agent 按规则取名、把用户的叫法（「e5」）映射到规范名（「ES-5」）并告诉用户对应关系。
- **mcp validate 层**：在 `topology_verify.rs` 里加一条校验——交换机名不以 SW 开头、端系统名不以 ES 开头就报错，挡住不规范命名落库。

要彻底解决「说 e5 建成 ES-6」这种，还得给 `node_add` 加 name 字段、定一套序号分配规则，让新增节点的名字不是 agent 随手拍的。

**属于：** 命名

**依据（看代码 + 实测）：** initialize 已经是 `SW-{n}`/`ES-{n}`（topology_compute.rs），但只在这条路；`node_add` 无 name 字段（topology_ops.rs L23-31），新增全靠 agent 自己取、序号自己挑；validate（topology_verify.rs）现在不校命名前缀。实测 A-C-03 说 e5 建成 ES-6。

**为什么值得做：** 命名对不上是会反复犯的一类错，根子在「名字和序号是 agent 临时拍的」。把规则钉死、再让 validate 兜底，就能根治一整类——既写进 skill（指引 agent 怎么取），又进 validate（不规范的挡在门外），两头都有保障。

**代价 / 要注意：** validate 加前缀校验是 Rust 小改；`node_add` 加 name 字段 + 序号规则是 MCP 改动，**得你点头**。要兼容现有数据（已经叫 SW-/ES- 的没问题，但要确认没有别的历史命名）。这条跟点子 1 的「修 imac 残余」一起做更顺。

**把握：** 高 · **改动：** 中（validate 校验小，node_add 加字段中）

### 8. 推理态交互：顶部 loading 下移到输入框，发送变终止，输入框不挡用户

**这是什么：** boss 要的：现在顶部那条全宽 loading banner 不明显，移除掉，把推理态的反馈放到输入框这一带——

- 推理中，输入框给个「正在推理中…」之类的提示文案，但**仍然能继续打字编辑**，不挡用户的主动操作（现在 textarea 本来就没禁用，保持）。
- 推理中，发送按钮变成「终止」按钮，点一下能停掉当前这轮推理。

这条拆成两步，因为代价差很多：

- **第一步（便宜，纯前端）**：移除顶部 `AgentRunStatusBar`，把推理态文案放进输入框区（动态 placeholder 或输入框旁一行小字），发送按钮在推理中先维持禁用或换个「推理中」视觉。这步不依赖任何新能力。
- **第二步（贵，要新建中断链路）**：让发送按钮真能「终止」。**现在根本没有终止 agent 的能力**——要从头建：后端加一个 Tauri 命令向 worker 进程组发信号并清掉运行 flag，前端加 `stopRun` action，按钮在推理中切成 Stop 形态点击触发。

**属于：** 交互

**依据（看代码）：** `AgentRunStatusBar` 在 App.tsx:386 顶部渲染；textarea 推理中不禁用但 Enter no-op、placeholder 不变（chat-pane/index.tsx:171/204）；发送按钮推理中 disabled、永远纸飞机图标（index.tsx:210）；终止能力前后端都没有，`run_claude_agent` 是 spawn_blocking + 单例锁、只有超时才 kill（commands.rs）。
