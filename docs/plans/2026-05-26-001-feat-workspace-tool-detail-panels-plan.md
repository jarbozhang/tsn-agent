# feat: 完善工作台工具详情面板

## 摘要

将“更新日志”从顶层页面迁入工作台的“设置”，以版本列表 + 详情的方式展示可给客户看的更新内容；将“Skill”和“执行日志”统一调整为左侧列表、右侧详情的主从布局。其中 Skill 支持查看具体 skill 的结构化详情，并提供受控编辑入口；执行日志点击某条记录后在右侧展示完整详情，不再把 JSON 内容折叠在每一行内部。

## 问题框架

当前工作台已经把“会话、执行日志、Skill、设置”收敛到内部工具栏，但仍存在三个体验断点：

- 更新日志仍是顶层页面，和“设置”这种面向应用信息、版本信息的入口不一致。
- Skill 面板只是静态说明卡片，无法查看单个 skill 的职责、阶段、状态，也没有编辑能力。
- 执行日志详情塞在每条日志行的折叠内容中，日志多时扫描困难，右侧空间没有被有效利用。

这次改动目标是完善信息架构和工具详情体验，不改变 TSN 分阶段工作流、规划器对接、仿真导出等业务主流程。

## 需求

- R1：更新日志放入“设置”工具中展示，不再作为独立顶层导航入口。
- R2：设置中的更新日志支持点击具体版本号查看该版本内容，默认选中当前应用版本或最新版本。
- R3：Skill 工具支持选择具体 skill 查看详情，包括名称、对应阶段、用途、状态、输入输出摘要和最近运行摘要等。
- R4：Skill 详情支持编辑。第一版应优先编辑受控的本地元数据或草稿配置，避免直接无保护地改写外部 `SKILL.md` 文件。
- R5：执行日志列表点击一条记录后，在右侧详情面板展示完整内容；列表行只保留摘要信息，不再内联展开 `<details>`。
- R6：执行日志详情继续使用现有脱敏后的日志数据，不展示未脱敏原文、凭证或大段原始工具输出。
- R7：三个工具面板在桌面端使用稳定的主从布局，在窄屏下退化为上下布局，避免文字挤压和控件重叠。
- R8：保持现有“会话、执行日志、Skill、设置”的工作台工具层级，顶部产品导航不再承担会话、日志或更新日志入口。

## 范围边界

本计划包含：

- 调整工作台工具信息架构。
- 迁移更新日志展示位置。
- 重构执行日志详情查看方式。
- 增加 Skill 列表、详情和受控编辑 UI。
- 补充相关单元测试和浏览器冒烟验证。

本计划不包含：

- 建设远程 skill 市场或插件安装系统。
- 直接在 UI 中任意编辑用户磁盘上的 `SKILL.md`、脚本或外部仓库文件。
- 改造真实 Agent skill 执行协议。
- 改变诊断日志持久化格式，除非为了选择态或详情展示必须做轻量补充。
- 改变 `CHANGELOG.md` 的生成逻辑和 release 分支发布规则。
- 修改 TSN 拓扑、时间同步、流量规划、INET 导出的业务数据结构。

## 技术决策

### D1：设置拥有版本和更新日志

将更新日志作为“设置”里的一个区域或子视图。设置面板左侧可以保留应用信息、规划器服务、存储位置等设置摘要，更新日志区域采用版本列表 + 详情区。

顶层导航可以收敛为仅保留“工作台”，或者在没有其他顶层页面时取消多余的页面切换控件，仅保留产品标题、版本号和运行状态。

### D2：详情面板统一采用主从布局

执行日志和 Skill 都采用相同的信息结构：

- 左侧：可扫描、可过滤或可选择的列表。
- 右侧：当前选中项的详情。
- 空状态：提示选择一条日志或一个 skill。
- 窄屏：列表在上，详情在下。

这样避免把每条记录都膨胀成局部展开块，也为后续“会话详情”“设置分组详情”留下统一交互模式。

### D3：Skill 编辑第一版做受控元数据编辑

“可编辑”不应直接等同于改写外部 skill 文件。第一版建议编辑应用内可控字段，例如：

- 显示名称。
- 描述。
- 启用状态。
- 备注。
- 默认参数草稿或提示词草稿。

这些编辑可以先保存在应用本地状态或本地持久化配置中。真实改写 skill 文件、校验 `SKILL.md` 结构、版本化 skill 变更可以作为后续能力。

### D4：执行日志详情只消费脱敏后的现有数据

右侧详情面板显示 `DiagnosticLogEntry` 的结构化字段和 `details`，不绕过现有日志仓库和脱敏流程。若需要更好的展示，可做格式化、分组和复制按钮，但不能扩大敏感信息暴露面。

## 风险与缓解

- 风险：工具抽屉宽度不足，主从布局显得拥挤。
  - 缓解：桌面端提高详情区最小宽度；窄屏使用上下布局；长文本使用换行和最大高度滚动。

- 风险：用户理解的“编辑 skill”是编辑真实 skill 文件。
  - 缓解：第一版 UI 文案明确为“编辑本应用中的 skill 配置/备注”；真实文件编辑能力另起计划，必须增加权限提示、结构校验和回滚机制。

- 风险：日志详情展示过多内容导致敏感信息泄漏。
  - 缓解：只使用现有脱敏日志对象；详情面板不读取原始运行输出；测试覆盖凭证脱敏。

- 风险：移除顶层更新日志后用户找不到版本信息。
  - 缓解：在设置入口附近保留当前版本号提示，并在设置中提供清晰“更新日志”分组。
