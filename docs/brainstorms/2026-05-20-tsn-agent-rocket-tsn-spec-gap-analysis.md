# TSN Agent 与箭载 TSN 技术规范差距分析

## Summary

本文档定义一份面向后续规划的差距分析：以 `docs/prototypes/箭载TSN技术规范_V1.2_1204-s.docx` 为规范目标，对照当前 TSN Agent MVP 的真实能力，识别从“入门拓扑生成器”走向“箭载 TSN 规范辅助设计工具”需要补齐的产品能力。

---

## Problem Frame

当前 TSN Agent 已经能从一句网络规模描述生成基础拓扑、单条控制流模板、最小 INET/OMNeT++ NED、`omnetpp.ini`、React Flow 展示数据、规划器输入和 manifest。它的定位仍是新手入门和项目文件生成，已在 README 和 NED 契约中明确不承诺完整 gPTP、TAS/GCL、CBS、FRER、PSFP 或真实业务流应用配置。

箭载 TSN 技术规范的目标明显更高：它面向箭载设备、系统和网络一致性，要求在静态配置、强时序调度、双平面冗余、容错时间同步、逐流监管、集中配置与运行监控等方面形成可互操作、可验证的工程基础。两者之间的差距不是单个导出字段缺失，而是产品能力层级不同。

这份分析用于明确差距边界，避免后续 `ce-plan` 直接从协议条款跳进实现细节，也避免把当前 MVP 的有意范围边界误判为实现 bug。

---

## Actors

- A1. TSN 新手用户：从自然语言开始生成拓扑和示例流，需要理解规范概念但不应被完整参数表淹没。
- A2. 箭载 TSN 方案设计者：需要评估拓扑、流量、同步、冗余和配置是否逐步接近规范要求。
- A3. TSN Agent：负责把用户意图、规范约束和当前项目状态转成可检查的模型与产物。
- A4. 规划器与仿真流程：消费 `flow_plan_1.json`、NED/INI 和后续调度/配置产物。
- A5. 后续实现/规划 agent：根据本文档拆分可执行功能，不再重新发明差距分类。

---

## Key Flows

- F1. 规范差距检查
  - **Trigger:** 用户已经生成或导入一个 TSN Agent 项目，希望知道它距离箭载 TSN 规范还差什么。
  - **Actors:** A2, A3
  - **Steps:** TSN Agent 读取 canonical project 和导出清单；按拓扑、流量、同步、整形、冗余、配置监控等维度检查覆盖情况；输出分级差距。
  - **Outcome:** 用户看到哪些能力已覆盖、哪些只是占位、哪些完全缺失，以及下一步优先级。
  - **Covered by:** R1, R2, R3, R4

- F2. 从 MVP 拓扑推进到规范化模型
  - **Trigger:** 当前项目需要进入下一阶段建模，不再只停留在基础拓扑和单条 ST 示例流。
  - **Actors:** A1, A2, A3, A4
  - **Steps:** 系统保留当前简单入口；在不强迫新手编辑参数表的前提下，逐步补齐流类型、VLAN/PCP、静态路径、同步域、GM、冗余路径和调度意图。
  - **Outcome:** canonical model 不必一次实现全部协议，但能承载后续规划器、INET 导出和规范检查所需的关键概念。
  - **Covered by:** R5, R6, R7, R8, R9

- F3. 后续规划交接
  - **Trigger:** 需要把本文档转入 `ce-plan` 或拆成多个实施阶段。
  - **Actors:** A5
  - **Steps:** 实现/规划 agent 读取本文档；按优先级选择第一阶段；把需求级差距转化为模型、UI、导出、测试和验证任务。
  - **Outcome:** 后续计划不需要再判断“该补什么”，只需要决定“怎么补、先补哪一层”。
  - **Covered by:** R10, R11, R12

## Requirements

**差距分析输出**

- R1. 差距分析必须区分当前已实现、部分实现、占位承诺和完全缺失，避免把 README 中明确延期的能力误报为缺陷。
- R2. 差距分析必须按规范能力域组织，至少覆盖拓扑与设备、流量模型、时间同步、流量调度、逐流监管、冗余容错、配置管理、运行监控和仿真/规划交付。
- R3. 每个差距必须说明对产品目标的影响，例如是影响新手解释、规划器输入、INET 仿真、规范一致性检查，还是未来硬件配置映射。
- R4. 差距分析必须给出优先级，不应只列问题清单。

**规范能力映射**

- R5. 后续 canonical model 需要从“基础拓扑 + 单条 ST 流”扩展为能表达规范核心概念的模型，包括 VLAN/VID、PCP、ST/AVB/BE 流类型、关键性、周期/偶发性、时延/抖动/带宽要求、路径、冗余意图和流标识。
- R6. 后续模型需要能表达箭载网络拓扑约束，包括双平面冗余、系统交换机不级联、局部环形/链式/无线 TSN、节点规模、跳数、链路速率和箭地互联边界。
- R7. 后续同步能力需要能表达静态 GM、时间同步树、多 gPTP 域、多 PTP 实例、ASed 容错时间选择和级间分离后的同步域关系。
- R8. 后续调度能力需要能表达 TAS/Qbv、CBS/Qav、队列映射、门控周期、AdminBaseTime、门控表容量、ST/AVB/BE 的调度和整形意图。
- R9. 后续安全隔离和可靠性能力需要能表达 PSFP/Qci、FRER/CB、逐流过滤、到达窗口、计量器、双平面路径、复制/消除窗口和异常流量默认丢弃策略。

**产品边界与演进**

- R10. 第一阶段不应直接实现完整 IEEE 协议栈或硬件下发；应优先补齐规范检查和后续导出所需的产品模型。
- R11. 下一阶段计划必须保持新手入口简单：规范参数应以模板、解释、默认值和检查结果逐步暴露，而不是一次性提供完整专家表单。
- R12. 后续交付应把“规范一致性检查”作为独立产品能力，而不只是隐藏在 NED/INI 导出里。

## Gap Analysis

**G1. 拓扑与设备建模：部分覆盖**

当前实现可以根据“交换机数量 + 每交换机端系统数量”生成线型交换机骨干和端系统接入拓扑，默认 1000 Mbps 链路，节点有稳定 ID、端口、位置、MAC/IP。规范中要求的双平面冗余拓扑、系统交换机不相互级联、关键单机双/三冗余、级间主干、局部环形/链式/无线 TSN、约 50 节点、≤6 跳、100M/1000M BE 自适应等，目前没有进入模型。

影响：当前拓扑足够做入门演示和最小 INET 加载，但不能用于判断箭载拓扑是否满足可靠性和静态配置原则。

优先级：P1。拓扑约束是后续同步、路径冗余和流调度的基础。

**G2. 流量类型与业务语义：部分覆盖**

当前实现固定生成 1 条 `ST` 控制流，包含源/目的、UDP 端口、PCP、周期、帧长、时延/抖动要求和手工路径。规范列出的飞控控制指令、周期消息、遥测、配置诊断、视频等多类流量，以及 `ST`、`AVB`、`BE` 三类流量的不同传输保证、关键性、丢包容忍和同步关系，目前没有被系统性表达。

影响：规划器输入有最小样例，但无法覆盖规范要求的业务流矩阵，也无法解释不同关键性等级的共网传输。

优先级：P1。流量语义直接决定 TAS、CBS、PSFP、FRER 的后续配置。

**G3. VLAN、流标识和队列映射：明显缺口**

当前 flow 只有 `pcp`，没有 `VID`、`DMAC+VID` 流标识、VLAN 帧意图、8 队列映射或 ST/AVB/BE 到队列的规则。规范明确要求网卡和交换支持 VLAN，并以 DMAC + VID 识别流，支持 8 个输出队列和优先级到队列映射。

影响：后续如果直接导出 TAS/CBS/PSFP，会缺少流识别和队列选择的规范依据。

优先级：P1。建议早于调度导出补齐。

**G4. 时间同步：仅占位**

当前 `simulationHints.timeSynchronization` 只有 `assumed-synchronized`，`omnetpp.ini` 明确说明 gPTP 后续扩展。规范要求 AS/ASed/ASdm 相关能力，包括两步法时间同步、静态 GM、静态时间同步树、多个 PTP 实例、多时钟域、FTTM 时间选择、容错时间同步和级间分离后的同步保持。

影响：当前系统无法生成、检查或解释任何真实时间同步配置，也无法支撑 TAS/PSFP 到达窗口的时基一致性。

优先级：P1。可以先做模型和检查，再做 INET 导出。

**G5. TAS/Qbv 与 CBS/Qav：仅文档承诺**

当前项目没有门控表、队列门状态、AdminCycleTime、AdminBaseTime、TickGranularity、CBS credit 参数或逐流/逐类整形模型。NED/INI 明确不配置 TAS/GCL、CBS、ATS。规范则要求网卡和交换同端口支持 TAS 与 CBS，所有队列支持 TAS，每端口至少 1024 个 TAS 门控表项，并支持逐类 CBS，网卡可选逐流 CBS。

影响：当前不能证明 ST 流有界时延、低抖动，也不能表达 AVB 带宽保证。

优先级：P2。应在时间同步和流/队列模型之后推进。

**G6. PSFP/Qci 逐流监管：缺失**

当前模型没有流过滤器、到达窗口、最大帧长过滤、计量器、非法 ST/AVB 默认丢弃或监控计数。规范要求交换支持至少 512 条流的逐流过滤，使用 PSFP 隔离异常流量，测速过滤虽可选但到达窗口和过滤能力是关键要求。

影响：当前不能分析异常设备、突发流量或非法流对关键业务的影响。

优先级：P2。对箭载可靠性和安全隔离很重要，但依赖流标识与同步/调度基础。

**G7. FRER/CB 冗余传输：只有规划器字段占位**

当前 planner exporter 固定输出 `redundant: "1"`，但 canonical model 没有双平面路径、复制流列表、消除流列表、R-TAG/SN、序列号窗口或帧消除超时配置。规范要求网卡处实现 FRER，支持复制与消除、向量恢复算法、SN 独立计数和不小于 8 的消除窗口。

影响：导出的 `redundant` 字段容易给人造成“已支持冗余”的误解，但实际缺少冗余路径和消除语义。

优先级：P1。至少需要先把“冗余意图”和“尚未可验证”区分清楚。

**G8. 集中配置模型：产品外壳接近，网络语义缺失**

当前应用具备会话、导出清单、文件写盘和本地 SQLite 工作台，和规范中的“地面网络控制器一次性下发配置、发射后断开”在产品形态上有相似点。但当前没有 CNC/CUC 语义、静态拓扑/流量规划参数集、NETCONF/SNMP 配置对象，也没有设备级可配置参数集合。

影响：当前可以交付文件，但不是规范意义上的网络配置生成器。

优先级：P3。短期可先把配置对象作为导出目标模型，不必做真实下发。

**G9. 运行监控和诊断：应用诊断已做，网络监控缺失**

当前诊断日志覆盖 Claude 交互、会话保存和导出文件刷新。规范要求的时间同步监测、TAS TransmissionOverrun、PSFP 通过/未通过计数、FRER 消除计数、FTTM 选择结果等网络运行监控参数没有模型或展示。

影响：当前诊断能帮开发者排查应用问题，但不能支持箭载网络运行状态监控或故障隔离。

优先级：P3。可作为后续“仿真结果/设备状态只读展示”能力推进。

**G10. 仿真交付：最小可加载，不是规范行为仿真**

当前 NED 使用 INET TSN 节点建立基础拓扑，`omnetpp.ini` 只保证可加载运行。规范相关的 gPTP、TAS、CBS、FRER、PSFP、业务流应用和规划结果回写都未生成。

影响：当前仿真只能验证拓扑结构，不验证确定性时延、抖动、冗余或隔离效果。

优先级：P2。应在模型补齐后逐步扩展 INI/配置导出。

## Acceptance Examples

- AE1. **Covers R1, R2, R4.** Given 当前项目只有基础拓扑和单条 ST 控制流 when 用户请求规范差距检查 then 输出必须标记拓扑、流量、同步、调度、冗余等能力的覆盖状态和优先级。
- AE2. **Covers R5, R6.** Given 用户生成“4 个交换机，每个交换机 5 个端系统” when 系统进入规范化建模阶段 then 模型必须能表示该拓扑是否只是线型骨干、是否缺少双平面冗余、是否满足跳数和链路速率约束。
- AE3. **Covers R7, R8, R9.** Given 一条控制流已存在 when 后续要求生成规范化配置 then 系统不得只依赖 `pcp` 和 `redundant` 字段推断完整 TAS、CBS、PSFP 或 FRER 配置。
- AE4. **Covers R10, R11.** Given 新手用户继续使用自然语言入口 when 引入规范能力 then UI 应通过模板、默认解释和差距检查逐步暴露复杂参数，而不是要求用户直接填写完整 IEEE 参数表。
- AE5. **Covers R12.** Given 规范检查发现 FRER 或 ASed 未实现 when 用户查看结果 then 系统应把它们显示为待补齐能力，而不是在导出文件中静默省略。

## Success Criteria

- 产品团队能明确区分当前 MVP、下一阶段模型扩展和长期设备配置/监控能力。
- 后续 `ce-plan` 可以直接按 P1/P2/P3 差距拆任务，不需要重新阅读整份 Word 规范才能确定第一阶段范围。
- 新手体验仍保持自然语言入口和渐进式解释，不因引入箭载规范而退化为专家参数编辑器。
- 后续导出的规划器输入、INET 配置和规范检查结果不再出现“字段看似支持、语义实际缺失”的误导。

## Scope Boundaries

- 本文档不要求当前实现立刻满足箭载 TSN 技术规范。
- 本文档不设计具体 TypeScript/Rust schema、组件结构或导出 JSON 结构。
- 本文档不实现或验证 IEEE 802.1AS、Qbv、Qav、Qci、CB 等协议。
- 本文档不覆盖真实硬件配置下发、NETCONF/SNMP 接入或设备互联互通测试。
- 本文档不把 Word 规范逐条转换为测试用例；它只提炼当前产品演进所需的关键差距。

---

## Key Decisions

- 差距分析按能力域组织：这比按章节复述规范更适合后续产品规划。
- P1 优先补模型基础：拓扑约束、流语义、VLAN/流标识、时间同步和冗余意图区分，是后续调度和导出的前置条件。
- 规范能力需要渐进暴露：面向新手的产品不能直接变成完整专家参数表。
- “规范一致性检查”应成为显式能力：它能在完整协议导出前提供价值，并降低误解当前 MVP 能力的风险。
